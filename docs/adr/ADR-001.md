# ADR-001 - Arquitetura fundacional do Ilex Logística

## Status

Proposto para revisão técnica na Sprint C.

## Data

2026-05-13

## Contexto

O projeto Ilex Logística precisa evoluir um MVP em ciclos curtos, com backend, frontend, infraestrutura, documentação e governança mantidos em repositórios separados. As entregas A-01, B-01 e C-01 criaram a base inicial de API FastAPI, Web Next.js e ambiente local via Docker Compose.

A Sprint C consolida a fundação técnica para reduzir risco operacional, melhorar onboarding de novos devs e criar rastreabilidade mínima para decisões, CI, QA e revisão de PRs.

## Decisão

Adotar uma arquitetura fundacional composta por:

- API em FastAPI com SQLAlchemy, Alembic e PostgreSQL.
- Web administrativo em Next.js, TypeScript e Tailwind CSS.
- Docker Compose no repositório `Infra` para ambiente local reproduzível.
- PostgreSQL como banco local padrão da stack.
- `.env.example` e templates específicos para padronização de variáveis locais.
- GitHub Actions para CI mínimo de API e Web.
- Templates de Issue e Pull Request no repositório `.github` para rastreabilidade Scrum.
- Documentação de arquitetura, riscos e QA mínimo no repositório `Docs`.

## Consequências positivas

- Onboarding local fica mais previsível com Compose, `.env.example` e documentação.
- API e Web passam a ter feedback automático de lint, testes ou build em PR.
- Responsabilidades por repositório ficam explícitas e revisáveis.
- Decisões arquiteturais passam a ter histórico versionado.
- Templates de Issue/PR reduzem perda de contexto, critérios e evidências.
- Plano de QA mínimo cria base objetiva para aceite da sprint.

## Consequências negativas

- Separação em múltiplos repositórios exige disciplina de sincronização entre API, Web, Infra, Docs e `.github`.
- Mudanças transversais podem exigir PRs coordenados em mais de um repositório.
- Docker Compose local não substitui validação de deploy real.
- `.env.example` precisa ser mantido atualizado sempre que novos contratos de configuração surgirem.
- CI mínimo não cobre todos os testes de integração e cenários end-to-end.

## Alternativas consideradas

| Alternativa | Benefício potencial | Limitação identificada |
|---|---|---|
| Monorepo único | Simplificaria PRs transversais e versionamento conjunto. | Aumentaria acoplamento e risco de mudanças cruzadas no início do MVP. |
| Cada dev configurar localmente sem Compose | Menos arquivos de infraestrutura no curto prazo. | Reduziria reprodutibilidade, onboarding e QA. |
| SQLite como banco local padrão | Setup inicial mais simples. | Não refletiria suficientemente o ambiente esperado com PostgreSQL e migrations. |
| CI apenas ao final da sprint | Menos esforço inicial de DevOps. | Aumentaria risco de regressões e retrabalho. |
| Kubernetes desde o início | Aproximação com ambientes mais robustos. | Complexidade operacional incompatível com a fase fundacional do MVP. |

## Alternativas rejeitadas

- Rejeitamos monorepo nesta fase porque o projeto já está organizado por responsabilidades e repositórios especializados.
- Rejeitamos ambiente local sem Docker Compose porque novos devs precisam subir a stack com comandos documentados e repetíveis.
- Rejeitamos SQLite como banco local padrão da stack porque PostgreSQL deve validar migrations e comportamento mais próximo do alvo.
- Rejeitamos adiar CI mínimo porque API e Web precisam de feedback rápido em PRs.
- Rejeitamos Kubernetes nesta etapa porque a prioridade da Sprint C é fundação simples, rastreável e validável.

## Impacto por repositório

### API

- Mantém backend FastAPI, contratos de configuração `ILEX_*`, migrations Alembic e testes básicos.
- Deve preservar `/health` para healthcheck local e CI.
- Deve manter comandos de lint/test compatíveis com GitHub Actions.

### Web

- Mantém frontend Next.js com `NEXT_PUBLIC_API_URL` como contrato público de configuração.
- Deve preservar lint e build como validações mínimas de PR.
- Deve documentar comandos locais equivalentes ao CI.

### Infra

- Centraliza Docker Compose, PostgreSQL local, `.env.example`, healthchecks, logs e observabilidade inicial.
- Deve manter templates de ambiente seguros para versionamento.
- Deve documentar startup, shutdown, diagnóstico e cleanup.

### Docs

- Centraliza ADRs, matriz de risco, plano de QA, cerimônias Scrum e materiais de revisão.
- Deve manter este ADR como referência fundacional para decisões futuras.

### `.github`

- Centraliza templates de Issue e Pull Request para tarefas, bugs, melhorias e revisão antes do merge.
- Deve exigir ID rastreável, critérios de aceite, plano de teste, evidências, riscos e checklist.

## Relação com a Sprint C

Este ADR registra as decisões que sustentam:

- C-01: Docker Compose base para API e DB.
- C-02: templates de variáveis e secrets locais.
- C-03: CI inicial da API.
- C-04: CI inicial do Web.
- C-05: healthchecks e logs básicos.
- C-06: templates de Issue e PR.
- C-08: matriz de risco técnico.
- C-09: plano de QA mínimo.
- C-10: cerimônias Scrum e governança semanal.

## Critérios de revisão

Antes do merge, revisar:

- Se as responsabilidades por repositório estão corretas.
- Se Docker Compose e PostgreSQL continuam sendo a escolha adequada para ambiente local.
- Se as variáveis documentadas refletem contratos reais de API e Web.
- Se CI mínimo cobre os comandos oficiais de lint, teste e build.
- Se riscos de múltiplos repositórios estão aceitos pela equipe.
- Se QA mínimo e templates Scrum são suficientes para rastreabilidade da Sprint C.
